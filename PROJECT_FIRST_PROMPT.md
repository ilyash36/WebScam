## 🎯 Цель системы документации

При работе над проектом мы используем **трехуровневую систему документации** для:
1. **Снижения стоимости запросов** - уменьшение output токенов за счет кэширования контекста
2. **Ускорения работы** - быстрый доступ к ключевой информации без повторного чтения всего кода
3. **Сохранения контекста** - важные решения и паттерны не теряются между сессиями
4. **Структуризации знаний** - четкое разделение философии, технических деталей и быстрого справочника

## 📁 Структура файлов документации

### 1. `PHILOSOPHY.md` (или `PHILOSOPHY.txt`)

**Что это**: Философия и концепция проекта - **ПОЧЕМУ** проект существует, какую проблему решает, каковы принципы работы.

**Содержит**:
- Главную идею проекта
- Проблему, которую решает проект
- Принципы работы и подходы
- Целевая аудитория и их потребности
- Ключевые правила и ограничения
- Тон коммуникации (если применимо)

**Как заполнять**:
- Создается **в начале проекта** - описывает концепцию до начала разработки
- Обновляется **при изменении концепции** - если меняется подход или добавляются новые принципы
- Должен быть **кратким и понятным** - 1-3 страницы максимум
- Фокус на **"ПОЧЕМУ"**, а не на "КАК"

**Пример структуры**:
```markdown
# Философия проекта [Название]

## Главная идея
[Краткое описание концепции - 2-3 предложения]

## Проблема, которую решает проект
[Описание проблемы пользователей]

## Принцип работы
[Как проект решает проблему - пошагово]

## Ключевые правила
[Важные ограничения и принципы]

## Целевая аудитория
[Кто будет использовать проект и почему]
```

### 2. `CLAUDE.md` (или `PROJECT.md`, `DOCS.md`)

**Что это**: Полная техническая документация проекта - **КАК** проект устроен, какие технологии используются, как работать с кодом.

**Содержит**:
- Контекст проекта (технологии, стек)
- Архитектуру и структуру проекта
- Технические требования
- Принципы разработки
- Паттерны кода и примеры
- Инструкции для AI-ассистента
- Глоссарий терминов
- Примеры хороших/плохих практик

**Как заполнять**:
- Создается **после начала разработки** - когда есть базовая структура
- Обновляется **постоянно** - при добавлении новых функций, изменении архитектуры
- Должен быть **подробным** - все важные технические детали
- Фокус на **"КАК"** - как писать код, как работать с проектом

**Пример структуры**:
```markdown
# Инструкции для AI-ассистента - [Название проекта]

## Контекст проекта
[Технологии, стек, версии]

## Философия проекта
[Ссылка на PHILOSOPHY.md или краткое изложение]

## Технические требования
[Архитектура, структура, зависимости]

## Принципы разработки
[Стиль кода, паттерны, правила]

## Структура проекта
[Описание директорий и файлов]

## Работа с БД/API/Другими компонентами
[Паттерны и примеры]

## Рекомендации для AI-ассистента
[Что делать/не делать при работе с кодом]
```

### 3. `.cursor/dev_cache.md`

**Что это**: Быстрый справочник для AI-моделей - **СЖАТАЯ** информация для мгновенного понимания контекста без чтения всего кода. **Также служит кэшем истории разработки**.

**Содержит**:
- Быстрый обзор (1-2 абзаца)
- Архитектура (структура проекта, принципы)
- Ключевые компоненты (краткое описание модулей)
- Важные детали реализации (особенности, ограничения)
- Часто используемые функции (примеры кода)
- Паттерны кода (типичные решения)
- **История разработки** (важные решения, текущий контекст, часто используемые паттерны)
- Ссылки на полную документацию

**Как заполнять**:
- Создается **после CLAUDE.md** - на основе полной документации
- **Обновляется постоянно** - при каждом значительном изменении или решении
- **Кэширует историю разработки** - важные решения, текущий контекст работы
- Должен быть **максимально сжатым** - только самое важное, без воды
- Фокус на **быстром доступе** - AI должен понять проект за 30 секунд чтения
- **КРИТИЧЕСКИ ВАЖНО**: Обновляй раздел "История разработки" при принятии важных решений или изменении контекста работы

## 🔄 Процесс работы с документацией

### ⚠️ ВАЖНО: При первом запросе пользователя

**НЕ создавай файлы документации сразу!** 

Сначала:
1. **ПРОЧИТАЙ** существующие файлы (если есть)

3. **ОБНОВИ** `.cursor/dev_cache.md` (если существует) - добавь/обнови раздел "История разработки"

### При начале нового проекта (если файлов еще нет):

**Это делается пользователем или AI по запросу пользователя, НЕ автоматически при первом запросе!**

1. **Создай `PHILOSOPHY.md`** (по запросу пользователя)
   - Опиши концепцию проекта
   - Определи проблему и решение
   - Установи ключевые принципы

2. **Создай `CLAUDE.md`** (после начала разработки, по запросу пользователя)
   - Опиши технический стек
   - Документируй архитектуру
   - Добавь примеры кода и паттерны

3. **Создай `.cursor/dev_cache.md`** (после CLAUDE.md, по запросу пользователя)
   - Сожми информацию из CLAUDE.md
   - Оставь только ключевые детали
   - Добавь примеры часто используемых функций
   - **Добавь раздел "История разработки"** для кэширования контекста

### В процессе разработки:

1. **При добавлении новой функции**:
   - Обнови `CLAUDE.md` - добавь описание функции, примеры использования
   - Обнови `.cursor/dev_cache.md` - добавь краткое упоминание в соответствующий раздел
   - **Обнови раздел "История разработки"** в `.cursor/dev_cache.md` - добавь информацию о новой функции

2. **При изменении архитектуры**:
   - Обнови `CLAUDE.md` - опиши изменения в разделе "Архитектура"
   - Обнови `.cursor/dev_cache.md` - обнови структуру проекта
   - **Обнови раздел "История разработки"** - зафиксируй архитектурное решение и его причины

3. **При изменении концепции**:
   - Обнови `PHILOSOPHY.md` - опиши новые принципы
   - Обнови `CLAUDE.md` - если концепция влияет на техническую реализацию
   - **Обнови раздел "История разработки"** - зафиксируй изменение концепции

4. **При принятии важных решений**:
   - **ОБЯЗАТЕЛЬНО обнови раздел "История разработки"** в `.cursor/dev_cache.md`
   - Зафиксируй решение, причины его принятия, альтернативы (если были рассмотрены)
   - Это поможет в будущем не повторять обсуждения и не тратить output токены

## 🎓 Принципы работы с документацией

### 1. Принцип иерархии информации

```
PHILOSOPHY.md (ПОЧЕМУ) 
    ↓
CLAUDE.md (КАК - подробно)
    ↓
.cursor/dev_cache.md (КАК - кратко)
```

- **PHILOSOPHY.md** - читается **один раз** в начале проекта для понимания концепции
- **CLAUDE.md** - читается **при необходимости** для понимания деталей
- **.cursor/dev_cache.md** - читается **каждый раз** для быстрого контекста

### 2. Принцип кэширования

**Цель**: Уменьшить расход output токенов при запросах к AI-моделям за счет кэширования контекста и истории разработки.

**Как работает**:
- AI-модель сначала читает `.cursor/dev_cache.md` (быстрый контекст + история разработки)
- Если нужно больше деталей - читает `CLAUDE.md` (полная информация)
- Если нужно понять концепцию - читает `PHILOSOPHY.md` (философия)

**Кэширование истории разработки**:
- `.cursor/dev_cache.md` содержит раздел "История разработки"
- В этом разделе хранятся важные решения, текущий контекст работы, часто используемые паттерны
- AI-ассистент **обновляет этот раздел** при принятии важных решений или изменении контекста
- Это позволяет **не повторять** уже известную информацию в output, снижая стоимость запросов

**Результат**: 
- Меньше повторений в output (история уже в кэше)
- Быстрее понимание контекста (вся ключевая информация в одном месте)
- Ниже стоимость запросов (меньше output токенов)
- Сохранение контекста между сессиями (история разработки не теряется)

### 3. Принцип структуризации

**Правило**: Каждый файл имеет свою роль и не дублирует информацию без необходимости.

- **PHILOSOPHY.md** - только концепция, без технических деталей
- **CLAUDE.md** - только технические детали, концепция кратко (или ссылка)
- **.cursor/dev_cache.md** - только ключевые факты, без объяснений

### 4. Принцип актуальности

**Правило**: Документация должна быть актуальной.

- Обновляй файлы **сразу** при значительных изменениях
- Не откладывай обновление документации "на потом"
- Если что-то устарело - либо обнови, либо удали

### 5. Принцип читаемости

**Правило**: Документация должна быть легко читаемой.

- Используй **заголовки** для структуры
- Используй **списки** для перечислений
- Используй **примеры кода** для иллюстрации
- Используй **эмодзи** для визуального разделения (опционально)
- Избегай **слишком длинных** абзацев

## 💡 Советы

1. **При первом запросе - ЧИТАЙ, не создавай** - сначала пойми контекст проекта через существующие файлы

2. **Кэшируй историю разработки постоянно** - обновляй раздел "История разработки" в `.cursor/dev_cache.md` при каждом важном решении

3. **Начинай с философии** - без понимания "ПОЧЕМУ" сложно правильно реализовать "КАК"

4. **Обновляй документацию сразу** - не откладывай, иначе забудешь важные детали

5. **Используй примеры** - код говорит громче слов

6. **Будь кратким в кэше** - `.cursor/dev_cache.md` должен быть максимально сжатым, но содержать актуальную историю разработки

7. **Будь подробным в документации** - `CLAUDE.md` должен содержать все важные детали

8. **Следуй структуре** - используй предложенные шаблоны для единообразия

9. **Помни о стоимости** - кэширование истории разработки снижает расход output токенов в будущих запросах
